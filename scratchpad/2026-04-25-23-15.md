# Title: Session continuity â€” security hygiene, staging deployment, and provider-agnostic context curation

## Issue Summary
Multi-day work session covering three connected initiatives:
1. **Security hygiene + cleanup** following the v2 simplification refactor.
2. **Staging instance** deployed alongside production on the same host so the v2 refactor could be validated against real traffic before cutover.
3. **Provider-agnostic context curation** â€” application-layer system that trims and summarises old tool results within a single agent loop when context grows beyond a threshold, vendor-neutral so it works for any current or future provider.

The curation work is not yet complete. As of writing it is implemented and deployed to staging but has not been observed firing in real tests; a diagnostic log has just been added to investigate.

## Root Cause
1. The simplification refactor exposed lingering dead config (`LLAMA_SSH_*` variables on prod, `.env.pre-revert.bak` backup file, internal LAN IPs in tracked docs and pipe valve descriptions) that weren't part of the codebase but lived in deployment artefacts.
2. The 50/50-cache-hit, multi-million-token requests on production weren't from "many tools" â€” they were from a small number of tools that returned huge payloads (web_scrape, deep_research) and stayed in context for every subsequent iteration of the tool loop. With `MAX_TOOL_ITERATIONS=10`, a single user turn could send 1.4M+ cumulative tokens for $1.29.
3. Two latent bugs surfaced during staging testing: tool execution time and tool results were never being persisted to the DB (placeholder `execution_time_ms: 0` and `tool_result: ''`), and hitting the iteration cap caused both providers to throw instead of returning whatever the model had gathered.

## Key Design Decisions

1. **Application-layer context management, not vendor-specific buttons.** Anthropic ships built-in `clear_tool_uses_20250919` (server-side context editing) and `compact_20260112` (server-side compaction) that would solve the same problem with one extra header. Rejected because the user wants future flexibility to use Mistral, llama.cpp, Alibaba, etc. Implementing curation in our own code gives us symmetry across providers, no vendor lock-in, full transparency for telemetry, and a pluggable summarizer model.

2. **Curator is provider-agnostic by working on the messages array before native transform.** Detects both Anthropic format (tool_result blocks within user-role content arrays) and OpenAI Responses format (top-level function_call_output items). Same algorithm runs on both. Adding a new provider only requires wiring `curatorConfig` + `onCurationEvent` into its `streamChat`.

3. **Summarisation, not dumb trimming, when context exceeds threshold.** User's preferred behaviour: instead of replacing old tool results with `[cleared, refetch if needed]` placeholders, use the configured `COMPACTION_PROVIDER + COMPACTION_MODEL` (Haiku by default) to write a 150â€“200-token summary that preserves facts, names, URLs, dates. The LLM still has the gist; only loses ability to quote raw text verbatim. Falls back to dumb placeholder only if the summarizer call fails.

4. **Reuse the existing compaction-model valves.** The pipe already has `COMPACTION_PROVIDER` / `COMPACTION_MODEL` / `ENABLE_COMPACTION` for between-turn compaction. The curator borrows the same configuration â€” no new valves needed in `owui-pipe.py`. User can later set those to point at any local llama.cpp / vLLM / Ollama / hosted Mistral that speaks an OpenAI-compatible API. Local-llama option deferred for now: ship Haiku ($1/M) as default, watch one month of cost data, reconsider if compaction spend is meaningful.

5. **Strict `keep_recent` policy.** Curator never trims the most recent N tool results regardless of size. A single huge first tool result will sit until iteration N+6 even if it's 100k tokens alone. Simpler, predictable, and the LLM's most recent context is always intact for the current step. Smarter per-tool TTL or LLM-driven culling are documented as upgrade paths if dumb trim proves insufficient.

6. **OpenAI dropped `previous_response_id` chaining.** Made `openai.js` symmetric with `anthropic.js` â€” both maintain full local input history, send it on every iteration. Slight wire-bandwidth penalty traded for: curator can manage messages, telemetry is uniform, and adding any future OpenAI-compatible provider is trivial.

7. **Iteration cap = graceful degradation, not error.** Hitting `MAX_TOOL_ITERATIONS` triggers one final tool-less call asking for a summary from gathered info. Cost: tiny extra token spend for the final call. Benefit: a $0.20+ tool-loop's work is never silently wasted. Final call's usage rolls into `totalUsage` so cost tracking stays accurate. `stop_reason: 'iteration_cap'` reported for telemetry.

8. **System prompt nudge enables safe trimming.** When more than one tool is enabled, the assembled system prompt appends a hint asking the agent to synthesise key findings between tool calls. The assistant's running narrative is never trimmed by the curator (only `tool_result` blocks are). So if the agent has summarised in its own assistant messages as it goes, even aggressive trimming of raw tool results doesn't lose information.

9. **Curation telemetry is first-class.** New `curation_events` table per-row, dashboard tile in the sidebar, dedicated `/api/stats/curation` route. The user explicitly asked for this so that we can see whether the system is helping and detect "thrashing" (same tool re-called after curation) â€” the latter not yet implemented but the data shape supports it.

10. **Lower threshold on staging for diagnostic value.** Production default `CURATION_TRIGGER_TOKENS=100000`; staging set to `30000` to catch realistic heavy requests in our test scenarios. Once we know the right value from real-world data, production will be tuned accordingly.
