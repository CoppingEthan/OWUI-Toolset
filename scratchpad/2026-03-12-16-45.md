# Title: Cost calculation bug fix, settings race condition fix, historical cost recalculation

## Issue Summary
Three related problems found during a cost audit comparing OWUI dashboard figures against Anthropic billing:
1. **Cost calculation model name mismatch** — `claude-sonnet-4-6` didn't match `claude-sonnet-4-5` in settings, causing costs to fall through to default $1/$3 pricing instead of $3/$15. All sonnet 4.6 costs were **3.3x too low**.
2. **Dashboard settings race condition** — API server and dashboard are separate processes with separate in-memory copies of the same SQLite file. When the dashboard saved a settings change, the API server's next `saveThrottled()` would overwrite it with its stale copy.
3. **No way to fix historical data** — Past requests stored incorrect costs with no mechanism to recalculate.

## Root Cause
1. `calculateCost()` in `server.js` uses `modelLower.includes(pattern)` to match pricing keys. `"claude-sonnet-4-6".includes("claude-sonnet-4-5")` is `false`, so it fell through to the default `{ input: 1.00, output: 3.00 }` instead of `{ input: 3.00, output: 15.00 }`.
2. `index.js` spawns API server and dashboard as separate child processes via `spawn()`. Both create their own `new DatabaseManager(dbPath)` with independent in-memory copies. The API server never calls `reload()` before `save()`, so dashboard writes are overwritten.

## Old Behaviour
- Sonnet 4.6 requests costed at $1/$3 per 1M tokens (default) instead of $3/$15
- Dashboard reported $19.42 for March 1-12 when Anthropic billed $72.25
- Settings edited via dashboard would disappear after ~1 second (overwritten by API server save)
- No way to recalculate historical costs after fixing pricing

## New Behaviour
- `claude-sonnet-4-6` and `claude-opus-4-6` pricing added to schema.sql defaults
- `calculateCost()` has a family-name fallback: if exact pattern match fails, tries matching by model family (opus, sonnet, haiku, gpt-5) — future-proof against version bumps
- `save()` in DatabaseManager now calls `_mergeExternalSettings()` first — checks if the disk file was modified since our last save, and if so, merges the settings table (using `updated_at` timestamps to keep the newer value)
- New dashboard button "Recalculate All Historical Costs" at `POST /api/settings/recalculate-costs` — re-processes every request_metrics row using current pricing and saves corrected costs

## Files Changed
- **src/database/schema.sql:93-100** — Added `claude-opus-4-6` ($5/$25) and `claude-sonnet-4-6` ($3/$15) pricing entries. Kept 4-5 entries for backward compatibility.
- **src/api/server.js:1433-1460** — Added family-name fallback matching to `calculateCost()`. If no exact pattern matches, tries matching model families (opus, sonnet, haiku, etc.) against settings entries.
- **src/database/database.js:23,69-107** — Added `_lastSaveMs` tracking and `_mergeExternalSettings()` method. Before each `save()`, checks disk mtime; if another process modified the file, merges their settings table into our in-memory copy using `updated_at` comparison.
- **src/dashboard/server.js:553-620** — New `POST /api/settings/recalculate-costs` endpoint. Reads all requests, recalculates cost for each using current pricing and cache multipliers, batch-updates the database.
- **src/dashboard/public/index.html** — Added "Recalculate All Historical Costs" button with status feedback in the settings panel.

## Key Design Decisions
- Family fallback matching (`opus`, `sonnet`, `haiku`) means future model version bumps (4-7, 4-8, etc.) will automatically match the closest settings entry without manual updates. Exact matches still take priority.
- Settings merge uses `updated_at` timestamps rather than "last write wins", so both processes can coexist safely without losing changes from either side.
- Recalculation endpoint duplicates the cost formula from `server.js` rather than importing it, because the two servers are separate processes. The formulas are identical.
